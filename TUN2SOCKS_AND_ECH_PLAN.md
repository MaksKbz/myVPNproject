# tun2socks (badvpn) — реализовано в v3.7. ECH — план на будущее

## 1. tun2socks (badvpn) — ✅ РЕАЛИЗОВАНО в v3.7 CIS-MAX

### 1.1 Что оказалось не так в предыдущем плане

Более ранняя версия этого документа утверждала, что нужен патч `--tunfd` в
badvpn (беря пример с ByeDPIAndroid/hev-socks5-tunnel). **Это было неточно.**
При реальном изучении исходников оказалось, что `badvpn/tuntap/BTap.h` уже
содержит нужный механизм "из коробки":

```c
enum BTap_init_type {
    BTAP_INIT_STRING,  // открыть /dev/net/tun по имени устройства (root)
#ifndef BADVPN_USE_WINAPI
    BTAP_INIT_FD,      // ← принять уже открытый fd напрямую!
#endif
};
```

`BTap_Init2()` с `BTAP_INIT_FD` — ровно то, что нужно: передать
`ParcelFileDescriptor.getFd()` из `VpnService.Builder.establish()` без
всякого патчинга. Отдельный патч `--tunfd` не понадобился.

### 1.2 Что было реально сделано

1. **Vendored badvpn** (не submodule, как и byedpi) в
   `app/src/main/jni/badvpn/` — обрезано с 12MB до ~4.6MB (убраны
   client/server/ncd/examples/dostest/win32-специфика, не нужные для
   tun2socks; список файлов взят из `compile-tun2socks.sh` — официального
   "быстрого Linux-билда" от автора badvpn).

2. **Патч `tun2socks.c`**: добавлен библиотечный режим
   `TUN2SOCKS_LIBRARY_MODE`, который:
   - оборачивает оригинальный CLI `int main(argc, argv)` в
     `#ifndef TUN2SOCKS_LIBRARY_MODE` (апстримный код НЕ тронут — сохраняет
     совместимость с будущими апдейтами badvpn);
   - добавляет `tun2socks_bridge_run(config)` / `tun2socks_bridge_stop()`
     (см. `tun2socks_bridge.h`) — переиспользует существующие
     `process_arguments()`, `terminate()` и весь event-handling код,
     заполняя `options.*` программно вместо `parse_arguments(argc, argv)`;
   - TUN открывается через `BTAP_INIT_FD` с `dup()`'нутым fd (оригинал,
     переданный из Kotlin, остаётся у `ParcelFileDescriptor` и закрывается
     Android-стороной как обычно);
   - остановка — через self-pipe, зарегистрированный в `BReactor` как
     обычный файловый дескриптор (тот же паттерн, что `BADVPN_USE_SELFPIPE`
     в `BUnixSignal.c`) — `BReactor` не защищён мьютексами (`NO_SYS=1`
     дизайн lwIP), поэтому вызвать `BReactor_Quit()` напрямую из другого
     потока небезопасно;
   - логирование через инжектируемый `Tun2SocksBridgeLogFunc`
     (`__android_log_print` на Android-стороне) вместо `BLog_InitStdout()`.

3. **`tun2socks_jni.c`** переписан: вместо пустого цикла `while(g_running)`
   вызывает `tun2socks_bridge_run()` в pthread, с fallback на прежнюю
   заглушку, если `BADVPN_AVAILABLE` не определён на этапе сборки.

4. **`CMakeLists.txt`**: `badvpn_tun2socks` собирается как статическая
   библиотека из ~50 .c-файлов (badvpn core + lwIP core, без api/apps/netif —
   не нужны при `NO_SYS=1`), с теми же флагами, что `compile-tun2socks.sh`
   (`BADVPN_USE_EPOLL`, `BADVPN_USE_SIGNALFD` — стандартные Linux/Bionic API,
   доступны на Android без изменений).

5. **`ciadpi_jni.c`**: включён `params.udp = true` — иначе SOCKS5 UDP
   ASSOCIATE (нужен для QUIC/DNS через tun2socks → `SocksUdpClient`)
   отклонялся бы byedpi на уровне `handle_s5()`.

### 1.3 Проверка

Полный путь **TUN → lwIP → BSocksClient → SOCKS5 → обратно** был
E2E-протестирован на этой машине (Linux x86_64):
1. Собран весь badvpn+lwIP набор + пропатченный `tun2socks.c` в
   `TUN2SOCKS_LIBRARY_MODE` — компилируется чисто, без предупреждений.
2. Создан реальный `/dev/net/tun`-интерфейс (`sudo`, т.к. локально нет
   Android-эмулятора), запущен `tun2socks_bridge_run()` в pthread.
3. Запущен минимальный Python SOCKS5-сервер (echo).
4. `nc 10.0.0.2 12345` отправил данные через TUN — фейковый SOCKS5-сервер
   получил `CONNECT` запрос и данные, эхо вернулось клиенту через тот же
   TUN. Оба направления (TUN→SOCKS5 и SOCKS5→TUN) подтверждены рабочими.
5. `tun2socks_bridge_stop()` корректно останавливает event loop без
   segfault/утечек (после исправления найденного и устранённого бага —
   `BLog_Free()` вызывал `free_func()` безусловно, для кастомного
   Android-логгера была нужна no-op-функция, а не `NULL`).

**Не проверено:** реальная сборка через Android NDK (недоступен в этой
рабочей среде — нет сети для скачивания NDK) и запуск на настоящем
Android-устройстве/эмуляторе. Компиляция под Bionic должна работать
идентично Linux x86_64 (те же `BADVPN_LINUX`/`BADVPN_USE_EPOLL` флаги,
никаких glibc-специфичных вызовов в задействованных файлах не найдено), но
это стоит подтвердить прогоном CI (`./gradlew assembleDebug`) и ручным
тестом на устройстве перед тем, как полагаться на этот путь в проде.

### 1.4 Известные ограничения после этой реализации

- ~~**IPv6 в ciadpi**: `params.ipv6 = false`~~ — **исправлено**. `ciadpi_jni.c`
  теперь настраивает `params.baddr` как dual-stack (`AF_INET6` +
  `in6addr_any`), `params.ipv6 = true`. `remote_sock()` в byedpi/proxy.c сам
  включает `IPV6_V6ONLY=0` на сокете назначения, когда семейство адреса
  назначения `AF_INET6` — этого достаточно, чтобы один и тот же baddr
  одинаково обслуживал и v4-mapped-v6 адреса (`map_fix(dst,6)` преобразует
  в них чистый IPv4 dst), и нативные IPv6-адреса. Проверено локально
  функциональным тестом, воспроизводящим точную логику `remote_sock()`/
  `map_fix()`: dual-stack сокет успешно подключается к обоим типам
  назначений и передаёт данные в обе стороны.
- **UDP fragmentation/MTU**: `tun2socks_bridge_run()` использует MTU=1500 по
  умолчанию; для мобильных сетей (LTE с меньшим MTU из-за инкапсуляции)
  может потребоваться подстройка — не протестировано на реальной мобильной
  сети.
- ~~**Split-tunnel self-exclusion bug**~~ — **исправлено**. В
  `BypassVpnService.runVpn()` раньше при непустом `allowedApps` вызывался
  `addAllowedApplication()`, а затем всё равно `addDisallowedApplication(
  packageName)` — Android бросал `UnsupportedOperationException` («Builder
  может иметь только allowed ИЛИ disallowed список, не оба»), которая молча
  проглатывалась. Теперь в allowed-режиме `addDisallowedApplication()` не
  вызывается вообще (`packageName` просто не входит в отфильтрованный
  allowed-список), а в disallowed-режиме (`allowedApps` пуст) оба вызова
  корректны, т.к. `addAllowedApplication()` в этой ветке не используется.

## 2. ECH (Encrypted Client Hello) — план остаётся актуальным

### 2.1 Почему это не "добавить флаг"

ECH — это не модификация ClientHello руками (как split/disorder в
PacketProcessor), а полноценная HPKE-операция:

1. Получить **ECH-конфигурацию** сервера — это TXT/HTTPS-запись DNS
   (`type65 HTTPS` с параметром `ech=...`) для домена. Наш `DohResolver`
   сейчас резолвит только A/AAAA — нужно добавить парсинг HTTPS/SVCB-записей
   через DoH (RFC 9460).
2. Сформировать **ClientHelloInner** (настоящий, с реальным SNI) и
   **ClientHelloOuter** (с публичным `public_name` вместо SNI) —
   зашифровать Inner через HPKE (`X25519 + HKDF-SHA256 + AES-128-GCM` — набор,
   который использует Chrome/Cloudflare) публичным ключом из ECH-конфига.
3. TLS-стек **должен уметь это делать сам** — вручную патчить байты
   ClientHello (как делает `PacketProcessor` для split) для ECH невозможно,
   потому что нужна полноценная HPKE-криптография и валидный TLS 1.3
   handshake state machine.
4. Единственный практичный путь на Android — использовать TLS-библиотеку с
   поддержкой ECH: **BoringSSL** (собранный с `--enable-ech`, как это делает
   Chrome/Cronet) — обычный Android `SSLEngine`/`javax.net.ssl` ECH не
   поддерживает. Значит: собрать `boringssl` под NDK (это тяжёлая, но хорошо
   документированная сборка, ~10-15 минут CI-времени), обернуть через JNI
   как `ssl_ech_jni.so`, и **весь TLS-трафик приложения должен идти через
   этот стек**, а не через системный `SSLEngine` — то есть фактически нужен
   пользовательский SNI-прокси (аналог `sslocal`/`v2ray`), который терминирует
   TLS от клиента... что для прозрачного VPN (TUN-режим, произвольные
   приложения) означает MITM внутри самого VPN-сервиса, а это отдельный
   архитектурный слой (TLS-интерцептирующий прокси на локальном сертификате),
   не совместимый с "трафик не покидает устройство и не расшифровывается"
   принципом текущего README.

### 2.2 Более реалистичная альтернатива для того же результата

Так как цель ECH — "скрыть SNI от DPI", а не именно ECH-протокол как таковой,
дешевле и быстрее по времени реализации (без BoringSSL-сборки) дать 80% эффекта:

- **Domain fronting через DoH HTTPS-записи** (получаем "front" домен с высокой
  репутацией, не блокируемый DPI, и коннектимся туда с реальным SNI внутри
  зашифрованного TLS record — актуально там, где DPI смотрит только на SNI,
  а не делает полный ECH-детект).
- **oobPosition/tlsRec фрагментация SNI-байтов** (уже реализовано в ciadpi
  через `-o`/`-r`, активно во всех текущих пресетах) — рвёт сигнатуру SNI
  без полноценного ECH.

Эти меры уже частично покрыты существующими пресетами (`oobPosition`,
`tlsRec`). Настоящий ECH стоит планировать отдельным релизом v4.0 с явным
выделением времени на сборку BoringSSL под NDK и тестирование на реальном
трафике.

### 2.3 Рекомендация

tun2socks (п.1), split-tunnel self-exclusion и IPv6 в ciadpi уже реализованы
и E2E-протестированы (в разной степени — см. таблицу ниже). Следующим
логичным шагом перед ECH является:
1. Реальная сборка через Android NDK (подтверждена зелёным CI-прогоном на
   GitHub Actions) + ручной прогон на реальном устройстве/эмуляторе —
   подтвердить, что путь работает не только на Linux x86_64 и в CI, но и
   при реальном использовании конечным пользователем.
2. Только затем — отдельная ветка `feature/ech` с CI-джобой, которая явно
   собирает BoringSSL (добавит 10-15 минут к каждому CI-прогону,
   нежелательно мешать с основной веткой, пока не стабилизировано).

## 3. КРИТИЧЕСКИЙ баг: краш всего приложения при рестарте tun2socks (v3.7.2)

### 3.1 Симптом (найден пользователем на реальном устройстве)

При первом реальном тесте на устройстве: приложение запускается, VPN
включается (foreground-уведомление появляется), но через несколько секунд
приложение "сворачивается" само по себе, как будто закрылось, а VPN при
этом не работает (уведомление может продолжать висеть какое-то время как
артефакт системы).

### 3.2 Причина

`tun2socks_bridge_run()` вызывает `BNetwork_GlobalInit()` и `BTime_Init()` —
обе функции унаследованы из апстримного badvpn и содержат:

```c
ASSERT(!bnetwork_initialized)   // BNetwork.c
ASSERT(!btime_global.initialized)  // BTime.h
```

В апстримном CLI-режиме tun2socks эти функции вызываются **ровно один раз**
за время жизни процесса (при старте, до `BReactor_Exec()`), поэтому проблема
никогда не проявлялась в оригинальном коде. Но в Android-интеграции
`tun2socks_bridge_run()` вызывается заново при **каждом** перезапуске
движка:

- `startAsnAutoDetect()` в `BypassVpnService.kt` — срабатывает через 1.5
  секунды после включения VPN, определяет оператора СНГ по ASN и, если
  активен пресет `"universal"`, делает `ProxyEngine.stop()` +
  `ProxyEngine.start()` для применения тюнингованного пресета;
- `switchToNextPreset()` — авто-переключение при провале connectivity-теста;
- любой ручной рестарт движка.

При **втором** вызове `bnetwork_initialized`/`btime_global.initialized` уже
равны 1 (никогда не сбрасываются обратно в 0 при остановке/teardown), и
`ASSERT` в badvpn (собранном без `NDEBUG`) вызывает `abort()`. Сигнал
`SIGABRT` убивает **весь процесс приложения** мгновенно — минуя все Kotlin
`try/catch` в `BypassVpnService`/`ProxyEngine` (они оборачивают только
JNI-вызовы с точки зрения Kotlin, но `abort()` в нативном коде завершает
процесс на уровне ОС, а не бросает исключение в JVM).

Именно поэтому пользователь наблюдал не Kotlin-краш с трассировкой стека, а
"тихое" сворачивание приложения: `logcat` в этом случае показывает
`Fatal signal 6 (SIGABRT)` в native-потоке (`tun2socks_jni`/бридж), а не
Java/Kotlin stacktrace.

### 3.3 Исправление

Обе функции сделаны идемпотентными — при повторном вызове просто
возвращаются без побочных эффектов (их тело физически не нужно выполнять
дважды: `BNetwork_GlobalInit()` лишь ставит `SIGPIPE` в игнор,
`BTime_Init()` лишь фиксирует точку отсчёта для относительных таймеров):

```c
// system/BNetwork.c
int BNetwork_GlobalInit (void) {
    if (bnetwork_initialized) { return 1; }
    ...
}

// system/BTime.h
static void BTime_Init (void) {
    #ifndef NDEBUG
    if (btime_global.initialized) { return; }
    #endif
    ...
}
```

### 3.4 Проверка

Написан и прогнан локальный C-репродюсер, in точности имитирующий Android
JNI-сценарий: `tun2socks_bridge_run()` → `tun2socks_bridge_stop()` →
`tun2socks_bridge_run()` (второй раз) на реальном `/dev/net/tun`.

- **До фикса**: второй запуск падал со `SIGABRT`
  (`system/BNetwork.c:54 Assertion failed`, exit code 134). После
  добавления фикса только для `BNetwork.c` обнаружился **второй** такой же
  баг — краш переместился на `system/BTime.h:76`. Оба исправлены.
- **После обоих фиксов**: стресс-тест из **5 циклов** rebuild-run-stop
  подряд (с интервалом 500мс, что даже агрессивнее, чем реальный
  1.5-секундный ASN auto-detect) — `EXIT_CODE=0`, ни одного краша, все 5
  циклов завершились строкой `"tun2socks bridge stopped cleanly"`.

**По факту следующего теста пользователя**: фикс раздела 3 сам по себе НЕ
устранил проблему полностью — приложение всё ещё крашится при нажатии
"Запустить VPN" (уже после появления foreground-уведомления, то есть до
истечения 1.5-секундной задержки ASN auto-detect). Значит есть ещё как
минимум одна причина краша, которая либо специфична для реального
Android/Bionic окружения (не воспроизводится в моём Linux x86_64
репродюсере), либо срабатывает раньше, чем я успел проверить. См. раздел 4.

## 4. CrashLogger — сборщик диагностики крашей без adb (v3.7.3)

### 4.1 Проблема диагностики

Пользователь тестирует APK на реальном устройстве без доступа к
`adb logcat` — соответственно, при возникновении краша единственная
доступная информация — визуальное наблюдение ("приложение свернулось").
Без текста ошибки/сигнала/стектрейса дальнейшая охота за багом вслепую
неэффективна: предыдущий фикс (раздел 3) был основан на гипотезе,
подтверждённой локальным репродюсером, но не на реальных логах устройства
— и оказался неполным.

### 4.2 Решение

Добавлен минимальный сборщик диагностики, работающий полностью на стороне
устройства без внешних инструментов:

- **`CrashLogger.kt`** — `Thread.setDefaultUncaughtExceptionHandler()`,
  перехватывает необработанные Kotlin/Java-исключения, дописывает
  стектрейс в файл `filesDir/crash_log_java.txt`, затем передаёт
  управление предыдущему обработчику (штатное поведение сохраняется).
- **`crash_handler.c`/`.h`** — нативный обработчик сигналов `SIGABRT`,
  `SIGSEGV`, `SIGBUS`, `SIGILL`, `SIGFPE`, `SIGTRAP` через `sigaction()`.
  Пишет диагностику (имя сигнала, `si_code`, faulting address, pid/tid) в
  файл `filesDir/crash_log_native.txt` **только через `write()`** — без
  malloc/printf/локалей внутри самого обработчика (это единственные
  async-signal-safe операции, доступные внутри сигнального контекста).
  После записи передаёт управление предыдущему обработчику (обычно
  системный `debuggerd`/`libsigchain`), чтобы стандартный
  crash-reporting/tombstone продолжал работать как раньше.
- **Контрольные точки** (`crash_log_checkpoint()`) расставлены перед
  каждым потенциально опасным нативным вызовом — `ciadpiStart()` перед
  `init_params()`/`run()`, `tun2socksStart()` перед `pthread_create`,
  `tun2socks_thread()` перед `tun2socks_bridge_run()`. Даже если краш
  происходит через сигнал, который наш обработчик не перехватывает
  (например, `SIGKILL` от low-memory killer — его вообще нельзя
  перехватить), по последней записанной строке в файле будет понятно,
  на каком этапе процесс остановился.
- **UI**: `MainActivity` при каждом запуске проверяет оба файла и, если
  есть данные, показывает их в заметной красной карточке в самом верху
  экрана — с кнопками "Копировать" (в буфер обмена) и "Закрыть" (удаляет
  логи). Не требует ADB — пользователь может скопировать текст и вставить
  прямо в чат.

### 4.3 Проверка

Функциональные тесты нативного обработчика (Linux x86_64, gcc):
- Намеренный `*(int*)NULL = 42` (`SIGSEGV`) — обработчик перехватил,
  записал `Signal: SIGSEGV (11)`, `faulting address: 0x0`, корректно
  передал сигнал дальше (процесс завершился стандартным механизмом,
  `exit code 139`).
- Намеренный `abort()` (`SIGABRT`) — аналогично, `Signal: SIGABRT (6)`,
  `exit code 134`.
- Полный сценарий: `crash_handler_install()` → TUN → `tun2socks_bridge_run()`
  → `stop()` — все контрольные точки записались в правильном порядке,
  чисто, без ложных срабатываний.

**По факту следующего теста пользователя (v3.7.3)**: лог краша содержал
ТОЛЬКО одну строку — "crash handler installed" (пишется при запуске
приложения, до нажатия кнопки), после неё абсолютно ничего: ни блока
`NATIVE CRASH`, ни Java-исключения, ни расставленных чекпоинтов
(`ciadpi_thread: entered`, `tun2socks_thread: entered`). Значит сам
CrashLogger не сработал корректно — раздел 5 разбирает найденные причины.

## 5. Фикс CrashLogger — почему лог был пустым (v3.7.4)

### 5.1 Баг №1: crash_handler.c компилируется отдельно в каждую .so

`crash_handler.c` — обычный `.c`-файл, добавленный в исходники ОБОИХ
модулей (`ciadpi_jni.so` и `tun2socks_jni.so`) в `CMakeLists.txt`. Это
означает, что после линковки у каждой `.so` — **свой собственный** набор
глобальных переменных (`g_log_fd`, `g_installed` и т.д.), полностью
независимый от другой библиотеки. `ProxyEngine.installCrashHandler()` в
v3.7.3 вызывал JNI-функцию, которая физически находится в `ciadpi_jni.so`
— соответственно:
- В `ciadpi_jni.so` файл лога открылся, `g_log_fd` валиден — строка
  "crash handler installed" (записанная именно оттуда) успешно попала в
  файл.
- В `tun2socks_jni.so` `crash_handler_install()` никогда не вызывался —
  `g_log_fd` там навсегда остался `-1`, и все `crash_log_checkpoint()`
  внутри `tun2socks_thread`/`tun2socksStart` молча ничего не писали (см.
  реализацию: `if (g_log_fd < 0 || !tag) return;`).

Если реальный краш происходит в tun2socks/badvpn-коде (что вероятно,
учитывая, где были найдены другие баги — см. раздел 3), вся диагностика
по этому пути просто терялась.

**Исправление**: добавлена вторая JNI-функция
`installCrashHandlerTun2socks()` в `tun2socks_jni.c`, вызывающая тот же
`crash_handler_install()`, но уже внутри области видимости
`tun2socks_jni.so`. `ProxyEngine.installCrashHandler()` теперь вызывает
обе JNI-функции.

### 5.2 Баг №2: SA_ONSTACK без реального sigaltstack()

`crash_handler_install()` указывал `sa.sa_flags = SA_SIGINFO | SA_ONSTACK`,
но **никогда не вызывал `sigaltstack()`** для регистрации самого
альтернативного стека. Если краш вызван переполнением стека (частый
сценарий в глубоко вложенном C-коде вроде lwIP — рекурсивные вызовы,
большие локальные буферы), обработчик сигнала попытался бы выполниться на
уже испорченном (переполненном) стеке текущего потока и сам немедленно
погиб бы — тихо, без единой записанной строки. Это отлично объясняет
полностью пустой лог: обработчик формально был "установлен", но не мог
физически исполниться при самом вероятном типе краша.

Дополнительная тонкость: `sigaltstack()` — **per-thread** настройка, не
глобальная для процесса. Обработчик устанавливался в главном UI-потоке
(`MainActivity.onCreate()`), но сами краши происходят в отдельных
`pthread`-потоках (`ciadpi_thread`, `tun2socks_thread`) — значит
недостаточно вызвать `sigaltstack()` один раз в UI-потоке, нужно делать
это в каждом рабочем потоке отдельно.

**Исправление**: добавлена функция
`crash_handler_install_altstack_current_thread()`, использующая
`__thread`-локальный буфер (TLS) под аварийный стек каждого потока.
Вызывается в начале `ciadpi_thread()` и `tun2socks_thread()`, до любого
потенциально опасного кода.

### 5.3 Проверка (контрольный эксперимент)

Написана тестовая C-программа с намеренным переполнением стека
(бесконечная рекурсия с локальным буфером) внутри отдельного `pthread`:

- **БЕЗ фикса** (`sigaltstack()` не вызван для потока — воспроизводит
  поведение v3.7.3): `SIGSEGV`, `exit code 139`, но лог-файл **абсолютно
  пуст** — ни одной записанной строки. Ровно то же поведение, что
  наблюдал пользователь на реальном устройстве.
- **С фиксом** (`crash_handler_install_altstack_current_thread()` вызван
  в начале потока): тот же `SIGSEGV`, `exit code 139`, но лог корректно
  содержит `Signal: SIGSEGV (11)`, `si_code: 2`, `faulting address`,
  `pid`/`tid`.

Также перепроверены исходные тесты из раздела 4.3 (nameренный
null-pointer dereference и `abort()`) — оба по-прежнему корректно
перехватываются после добавления `sigaltstack()`.

**Не проверено**: сборка через реальный Android NDK (см. CI) и реальный
краш на устройстве — логика идентична для Linux/Bionic (`sigaltstack()`
и TLS через `__thread` — стандартные POSIX/Bionic-функции, без изменений
на Android), но подтверждение появится только после того, как
пользователь пришлёт содержимое карточки краша из следующего теста.

## 6. SIGKILL-гипотеза и checkpoint-трассировка (v3.7.5)

### 6.1 Новые данные от пользователя (устройство OPPO/ColorOS)

После установки v3.7.4 (с исправленным CrashLogger) пользователь сообщил:
- Лог краша теперь содержит ОБЕ строки установки обработчика:
  `ciadpi_jni: crash handler installed` И
  `tun2socks_jni: crash handler installed` — то есть баг №1 из раздела 5
  (установка только в одной .so) действительно исправлен, оба модуля
  готовы перехватывать сигналы.
- После этих строк — снова полная тишина: ни блока `NATIVE CRASH`, ни
  Java-исключения.
- Устройство: OPPO (ColorOS). Приложение исчезает **мгновенно на рабочий
  стол БЕЗ системного диалога** "Приложение остановлено" / "App has
  stopped".
- Настройки батареи для приложения уже стоят "Без ограничений" —
  стандартная оптимизация исключена как причина.

### 6.2 Пересмотр диагноза: SIGKILL

Если бы это был непойманный Kotlin-краш, Android показал бы системный
диалог. Если бы это был native-краш (`SIGABRT`/`SIGSEGV`/...), теперь
корректно установленный обработчик (раздел 5) должен был бы его
перехватить. Ни то, ни другое не произошло — при этом приложение
**мгновенно исчезает без каких-либо следов**.

Это классическая картина **`SIGKILL`** — сигнала, который система (или
прошивка) посылает процессу напрямую через ядро. `SIGKILL` **невозможно
перехватить никаким кодом** — ни `sigaction()` в C, ни
`Thread.UncaughtExceptionHandler` в Java/Kotlin, потому что ядро Linux
принудительно завершает процесс, не давая ему шанса выполнить
какой-либо обработчик.

ColorOS (прошивка OPPO) традиционно известна одним из самых агрессивных
среди Android-прошивок управлением фоновыми процессами — отдельно от
стандартных настроек "Оптимизация батареи" (которые уже проверены и
установлены как "Без ограничений") там есть собственный механизм
"Автозапуск" (Startup Manager / Phone Manager → Privacy Permissions →
Startup Manager), который может принудительно завершать процессы
приложений, не входящих в белый список, особенно VPN-приложений не из
официального AppGallery/Play Store.

### 6.3 Решение: checkpoint-трассировка без нативных сигналов

Поскольку `SIGKILL` невозможно перехватить, единственный способ понять,
**на каком именно шаге** процесс был убит — расставить частые
контрольные точки, которые пишутся на диск СРАЗУ (не буферизуются), и
после следующего запуска посмотреть, какая точка была записана
последней. Добавлено:

- `CrashLogger.checkpoint(context, tag)` — использует
  `RandomAccessFile.fd.sync()` для немедленного сброса на диск (не
  полагается на `File.appendText()`, которая теоретически может быть
  отложена буферизацией ОС), с ограничением размера файла (64 КБ) чтобы
  не расти бесконечно между сеансами без крахов.
- Чекпоинты расставлены по всему пути от нажатия кнопки до полного
  запуска VPN: `MainActivity` (нажатие "Запустить VPN" →
  `VpnService.prepare()` → `onActivityResult` → `startService()`),
  `BypassVpnService` (`onCreate` → `onStartCommand` → `startVpn` →
  `startForeground()` → `runVpn()` → `builder.establish()` →
  `ProxyEngine.start()` → `isNativeTun2socksBuilt()` →
  `runNativeIdleLoop()`).
- Также добавлены системные колбэки `onTaskRemoved()`, `onLowMemory()`,
  `onTrimMemory()` в `BypassVpnService` — если система убивает сервис из
  экрана "Недавние" или из-за нехватки памяти, это будет видно явно и не
  спутается с крахом непосредственно от нажатия кнопки.
- UI: карточка краша в `MainActivity` теперь показывается, даже если оба
  крашлога (Java и Native) пусты, но есть checkpoint-лог — именно этот
  сценарий актуален для `SIGKILL`. Показываются последние 4000 символов
  (а не первые) — самое важное всегда в конце, перед смертью процесса.

### 6.4 Ожидаемый результат следующего теста

Если гипотеза `SIGKILL` верна, checkpoint-лог покажет последнюю
успешно выполненную точку — например, если она обрывается сразу после
`"startVpn: перед startForeground()"`, это укажет на несовместимость
`foregroundServiceType="specialUse"` с ColorOS; если обрывается после
`"runVpn: перед builder.establish()"` — проблема в самом
VPN-туннелировании; если процесс убивается ДО первого чекпоинта вообще
(лог полностью пуст) — это подтвердит, что ColorOS убивает процесс ещё
на старте, до выполнения кода приложения (например, через политику
"не разрешён автозапуск").

В любом из этих случаев программное решение невозможно (нельзя
перехватить/предотвратить `SIGKILL` от системы) — единственный выход:
инструкция для пользователя по добавлению приложения в список разрешённых
в ColorOS Startup Manager / отключению агрессивной оптимизации именно
для этого приложения.
