# myVPNproject v3.7 CIS-MAX — Android DPI Bypass

Android-приложение для обхода DPI-блокировок без root и без внешнего сервера.  
Трафик перехватывается локально через TUN-интерфейс, форвардируется полноценным
нативным tun2socks (badvpn/lwIP) в SOCKS5, где нативный движок ciadpi (byedpi)
делает DPI-обход (TCP split/disorder/fake).

Вдохновлено проектами
[ByeDPI (ciadpi)](https://github.com/hufrea/byedpi),
[ByeDPIAndroid](https://github.com/dovecoteescapee/ByeDPIAndroid),
[badvpn/tun2socks](https://github.com/ambrop72/badvpn) и
[Zapret](https://github.com/bol-van/zapret).

## Архитектура v3.7 CIS-MAX

```
Браузер/приложение
       ↓
   TUN-интерфейс (10.0.0.2/24)    ←— BypassVpnService.kt
       ↓
  tun2socks  (badvpn + lwIP, C, JNI)     ←— tun2socks_jni.so
  Читает/пишет TUN fd НАПРЯМУЮ в нативном коде (dup()'нутый fd),
  полноценный TCP/IP стек lwIP терминирует TCP-соединения,
  UDP (QUIC/DNS) идёт через SOCKS5 UDP ASSOCIATE
       ↓
   SOCKS5  127.0.0.1:1080
       ↓
  ciadpi  (byedpi, C, JNI)              ←— ciadpi_jni.so
  TCP split / disorder / fake / OOB / TLS record split
       ↓
     Интернет
```

**Важно (история v3.6→v3.7):** до этой версии `PacketProcessor.kt` фрагментировал
TLS ClientHello и писал оба фрагмента **обратно в тот же TUN fd** — но у Android
TUN-интерфейса нет автоматической маршрутизации/NAT наружу: всё, что пишется
обратно в TUN, интерпретируется системой как **входящий** трафик для этого же
интерфейса, а не отправляется на реальный сервер. Реальная доставка пакетов в
интернет требует либо отдельного `protect()`-нутого сокета на каждое TCP/UDP
соединение, либо полноценного userspace TCP/IP стека поверх такого сокета —
именно эту роль и выполняет tun2socks (lwIP). Начиная с v3.7 это реализовано:
`tun2socks_bridge_run()` (патч в `tun2socks.c`) принимает уже открытый TUN fd
через `BTAP_INIT_FD` (не требует root) и полностью форвардит TCP через lwIP →
`BSocksClient` → SOCKS5, а UDP — через `SocksUdpClient` UDP ASSOCIATE.
Путь TUN → lwIP → SOCKS5 → обратно протестирован E2E на Linux (реальный TUN +
тестовый SOCKS5-сервер) в рамках разработки.

Пока native tun2socks активен, Kotlin **не читает TUN fd** — старый
`PacketProcessor`-цикл остаётся только как fallback для stub-сборки без badvpn
(см. `ProxyEngine.isNativeTun2socksBuilt()`).

### Компоненты

| Компонент | Роль |
|---|---|
| `BypassVpnService.kt` | TUN-интерфейс, foreground service, передаёт `tunFd` в `ProxyEngine`; выбирает native/fallback путь |
| `ProxyEngine.kt` | Kotlin JNI-мост: запуск ciadpi → tun2socks, `isNativeTun2socksBuilt()` |
| `ciadpi_jni.c` | C JNI-мост для byedpi (реальный, vendored, не submodule) |
| `tun2socks_jni.c` | C JNI-мост для badvpn/tun2socks (реальный, vendored) |
| `badvpn/tun2socks/tun2socks_bridge.h` | Библиотечный API поверх апстримного tun2socks.c (принимает открытый fd вместо CLI) |
| `ConfigManager.kt` | Пресеты (включая СНГ: kz-telecom/mts-ru/beeline-ru/rostelecom), JSON-сериализация конфигурации |
| `PacketProcessor.kt` | Fallback-путь (stub-сборка без badvpn): TCP split TLS ClientHello (IPv4 + IPv6) |
| `DnsInterceptor.kt` | Перехват UDP:53 на уровне TUN → резолв через `DohResolver` → синтез DNS-ответа |
| `DohResolver.kt` | DoH-клиент на чистом `HttpsURLConnection`/`org.json` (без внешних зависимостей) |
| `NetworkProfileDetector.kt` | Авто-определение оператора СНГ по ASN (ip-api.com через `protect()`-сокет) |
| `ChecksumUtils.kt` | IPv4/IPv6 IP/TCP/UDP чек-суммы, общие для PacketProcessor и DnsInterceptor |
| `MainActivity.kt` | UI: Jetpack Compose — пресеты, приложения, справка |

## Сборка

byedpi и badvpn/tun2socks/lwIP **вендорены прямо в репозиторий** (не submodule) —
достаточно обычного клона:

```bash
git clone https://github.com/MaksKbz/myVPNproject.git
cd myVPNproject
./gradlew assembleDebug
```

Собранный APK содержит реальный DPI-обход (ciadpi) и реальный TUN→SOCKS5
форвардинг (tun2socks) — не stub. Если badvpn-исходники по какой-то причине
отсутствуют в дереве (`app/src/main/jni/badvpn/tun2socks/tun2socks.c` не
найден), CMake автоматически переключается в safety-fallback: `tun2socks_jni.so`
собирается без реального badvpn, а `BypassVpnService` использует
Kotlin `PacketProcessor`-цикл как fallback (см. предупреждение выше про его
ограничения).

### Требования

- Android Studio Hedgehog+
- JDK 17+
- Android SDK API 24+
- **NDK 27.0.12077973** (устанавливается автоматически через CI или `sdkmanager`)

      в badvpn защищены апстримными `ASSERT(!initialized)` — при **первом**
      запуске `tun2socks_bridge_run()` всё в порядке, но эти функции
      вызываются заново при **каждом** перезапуске движка

      `preset auto-switch`, ручной рестарт), а глобальные one-shot флаги
      никогда не сбрасываются обратно в 0. Второй вызов → `ASSERT` →
      `abort()` → `SIGABRT` убивает **весь процесс приложения** мгновенно,
      минуя все Kotlin `try/catch`. Исправлено: обе функции сделаны
      идемпотентными. Воспроизведено и подтверждено локальным
      C-репродюсером (5 циклов restart подряд, `EXIT_CODE=0`, без единого
      краша, после фикса).

      есть ещё как минимум одна причина, не покрытая этим фиксом, и не
      воспроизведённая пока на Linux x86_64 (либо специфична для реального

      периодический тик каждую секунду внутри `runNativeIdleLoop()`,
      чтобы резко сузить 5.3-секундное окно неопределённости.

      входа в `runNativeIdleLoop()`. Между этой точкой и следующим

      сработали ОБА ожидаемых следующих чекпоинта

      `AsnAutoDetect: вызываем detect()` через ~1.5с), но **ни один из
      них не появился**. Значит процесс умер **менее чем за секунду**
      после входа в рабочий цикл — раньше, чем ASN auto-detect вообще
      успевает проснуться от `sleep(1500)`. Это **исключает ASN
      auto-detect** как причину. Правдоподобная новая гипотеза: сам

      `lowMemory=true` непосредственно перед обрывом лога.

      203 (+20) за 255мс между `mem-watch #1` и `#2`, затем стабилизировался
      на 203, а смерть процесса произошла в окне ~250-500мс ПОСЛЕ этого
      (между `mem-watch #3` и ожидаемым, но отсутствующим `#4`). Это
      указывает на резкий приток ~20 параллельных соединений (реальные
      приложения телефона, установившие сессии в момент включения VPN) —
      краш происходит не из-за нехватки ресурсов, а, вероятно, из-за
      race condition в самом C-коде (tun2socks/lwIP/ciadpi), проявляющегося
      именно при такой параллельной нагрузке (не воспроизведённой
      синтетическим E2E-тестом с одним TCP-соединением). Добавлены
      нативные чекпоинты: `tun2socks: TCP client #N accepted` (в
      `listener_accept_func()`, badvpn/tun2socks) и
      `ciadpi: SOCKS5 client fd=N accepted` (в `on_accept()`, byedpi) —
      теперь при следующем крахе будет видно точное количество соединений,
      обработанных к моменту смерти процесса, что должно помочь
      локализовать race condition (например, переполнение внутреннего
      пула `MEMP_NUM_TCP_PCB`/`DEFAULT_UDPGW_MAX_CONNECTIONS=256`, либо
      конкретный N-й клиент, вызывающий баг).
- [x] **v3.7.9: НАЙДЕН И ИСПРАВЛЕН критический баг САМОЙ ДИАГНОСТИКИ —
      `CrashLogger.clearAll()` терял native crash-лог через `unlink()`
      при открытом `fd`** — пользователь прислал четвёртый checkpoint-лог
      (после установки v3.7.8 с нативными connection-чекпоинтами).
      Нативная (C) часть лога состояла ровно из 2 строк
      (`"ciadpi_jni: crash handler installed"`,
      `"tun2socks_jni: crash handler installed"`) — НИ ОДНОГО из новых
      connection-чекпоинтов, НИ ОДНОГО маркера входа в `ciadpi_thread`/
      `tun2socks_thread`, при том что Kotlin-сторона лога чётко
      подтверждает: оба потока успешно стартовали
      (`ProxyEngine.start() вернул true`) и процесс прожил ещё 700+ мс.
      Анализ таймстампа `t=1783770713` (секунды) показал, что обе строки
      "installed" были написаны **новым процессом при автоперезапуске
      после краха**, а не тем, что упал — native crash-лог на момент
      самого падения был пуст. Найдена причина: `crash_handler.c`
      открывает `g_log_fd` один раз на всё время жизни процесса, а
      старая версия `CrashLogger.clearAll()` (кнопка "Закрыть" на
      карточке краша) удаляла (`File.delete()` = `unlink()`) этот же
      файл, пока `fd` был всё ещё открыт тем же процессом — типичный
      сценарий использования: пользователь видит карточку краша, жмёт
      "Закрыть", тут же снова жмёт "Запустить VPN" в рамках того же
      процесса (Activity не пересоздаётся). После `unlink()` при
      открытом `fd` все последующие записи `crash_log_checkpoint()`
      из `ciadpi_thread`/`tun2socks_thread` успешно проходят системный
      `write()`, но пишутся в недостижимый по пути inode — и
      безвозвратно теряются при смерти процесса. Подтверждено локальным
      C-экспериментом (`unlink()` при открытом `fd` → данные недостижимы
      по пути; `truncate(path, 0)` при открытом `fd` → данные корректно
      видны). Исправлено: `clearAll()` теперь обрезает
      `NATIVE_CRASH_FILE` до нуля по тому же пути
      (`RandomAccessFile.setLength(0)` → `ftruncate()`) вместо удаления —
      `fd`, уже открытый в C-коде, продолжает указывать на тот же файл.
      Все предыдущие гипотезы (OOM/LMK, ASN auto-detect, SIGKILL) были
      построены на данных, где native crash-лог был систематически
      пуст из-за этого бага — сейчас, после фикса, впервые появится
      шанс увидеть реальный сигнал/стектрейс или connection-чекпоинты,
      если крах вообще перехватываем (не чистый `SIGKILL`).
      См. подробности: `TUN2SOCKS_AND_ECH_PLAN.md`, раздел 10.
- [x] **v3.7.10: НАЙДЕНА И ИСПРАВЛЕНА РЕАЛЬНАЯ ПРИЧИНА КРАША — SIGSEGV
      из-за неинициализированного `params.mempool`** — фикс v3.7.9
      сработал: пользователь прислал лог, где впервые появилось РЕАЛЬНОЕ
      содержимое native crash-лога:
      ```
      tun2socks: TCP client #4 accepted
      ciadpi: SOCKS5 client fd=176 accepted
      Signal: SIGSEGV (11)
      si_code: 1
      faulting address: 0x2
      ```
      Произошло дважды подряд (до и после автоперезапуска) с идентичным
      `faulting address: 0x2` — полностью детерминированный баг, не
      зависящий от таймингов/гонки потоков. `0x2` — характерный адрес:
      NULL + 2 байта. Локальный эксперимент подтвердил:
      `offsetof(struct mphdr, cmp_type) == 2` — точное совпадение.
      Найдена причина: byedpi использует глобальный кэш desync-параметров
      по IP назначения (`cache_get()`/`cache_add()` в `extend.c`,
      обращаются к `params.mempool->cmp_type`), который в апстримном
      коде создаётся через `params.mempool = mem_pool(MF_EXTRA,
      CMP_BITS)` в `parse_args()`. Наш `ciadpi_jni.c` не использует
      `parse_args()` — он строит `struct params` вручную в собственной
      `init_params()`, которая никогда не инициализировала
      `params.mempool` (оставался `NULL` с версии v3.7). Как только
      обработка соединения доходит до пути, использующего кэш (не на
      первом соединении — зависит от внутренней логики
      `find_dp()`/`on_trigger()`, отсюда и падение именно на 4-м
      клиенте), происходит разыменование `NULL->cmp_type` → `SIGSEGV`
      по адресу `0x2`. Подтверждено локальным C-репродюсером с реальным
      `sigaction()`-обработчиком — получен **точно такой же** вывод
      (`SIGSEGV (11)`, `si_code: 1`, `faulting address: 0x2`), как на
      реальном устройстве, и подтверждено, что после фикса
      (`mempool = mem_pool(...)` перед стартом, `mem_destroy()` при
      остановке движка) тот же код отрабатывает без падения.
      Обе главные гипотезы предыдущих версий (OOM/LMK, race condition
      под параллельной нагрузкой) были ошибочными — причина существовала
      с самого начала проекта и была простой забытой инициализацией,
      которую невозможно было увидеть раньше из-за отдельного бага
      диагностики (v3.7.5-v3.7.8), исправленного только в v3.7.9.
      См. подробности: `TUN2SOCKS_AND_ECH_PLAN.md`, раздел 11.

      подключён". Причина: старый `stopVpn()` вызывал `ProxyEngine.stop()`
      (делает `pthread_join()` на обоих нативных потоках) ДО закрытия
      `vpnInterface`, а системный VPN-индикатор Android привязан именно к

      `resolve()` использует блокирующий `getaddrinfo()` без таймаута
      прямо в event loop `ciadpi_thread` — если в момент остановки шёл
      DNS-резолв, `pthread_join()` в главном потоке мог ждать неопределённо
      долго, откладывая закрытие `vpnInterface`. Исправлено: `vpnInterface`
      теперь закрывается ПЕРВЫМ ДЕЛОМ (немедленно гасит индикатор), а
      `ProxyEngine.stop()` перенесён в отдельный daemon-поток (не блокирует
      ничего). Добавлена защита от гонки при быстром цикле
      "Остановить"→"Запустить" — `startVpn()` дожидается (с таймаутом 3с)
